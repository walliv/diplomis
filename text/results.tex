\chapter{DMA Iuventus}
\label{chap:dma_iuventus}

Since this thesis proposes a solution that can access the filesystem located on an NVMe device
shared between the host OS and an FPGA accelerator, a specific IP core is needed on the
accelerator's side. Without this, the user logic inside an FPGA would deal with raw PCIe data not
only from the non-volatile storage but from the queue entries as well. Therefore, the \emph{DMA
Iuventus} IP core has been developed to process requests between an accelerator and an NVMe
device\cite{iuventus_github}. The term "DMA" has been retained to highlight the mechanism of the
direct copying of data to the adjacent storage without a host CPU facilitating such transfers (this
transfer in a \ac{P2P} manner is sometimes called \emph{Host bypassing}). This IP sits in the FPGA
on the integrated block for PCIe to accept NVMe-specific traffic alongside logic that processes
transactions to \ac{C/S} registers\footnotemark.

\footnotetext{The source files of the module are available in the cited repository under the
    \path{comp/dma/dma_iuventus} directory}

\section{TLP Headers}

The DMA IP utilizes 3 of the 4 interfaces provided by the integrated block for PCI Express, namely
\ac{RQ}, \ac{CQ}, \ac{CC}. Each one of these uses its own template of the TLP header that the DMA
core needs to create or process. The \ac{RQ} header has a size of 4~DW and its format is depicted
in~\cref{fig:pcie_tlp_rq_hdr}. The \emph{Address} field is aligned with respect to DWords since its
two bottom bits are reserved\footnotemark. For \acp{MWr}, the \emph{Dword count} field specifies the
size of the payload following the header, whereas, for \acp{MRd}, it specifies the required amount
of data that a Completer needs to send in its Completion. The type of a request is defined in the
\emph{Req type} field. The \emph{Func} field informs the PCIe IP which Function generated the
request (more on that later)\cite{pcie_amd_ip}. The last used field is the \emph{NoSnoop} bit, which
instructs the host CPU that the underlying address space targeted by the transaction does not
require hardware-enforced cache coherency\cite{pcie_spec}. This means that the address space is not
shared by any other device in the system. The \ac{RQ} interface is used to dispatch \ac{SQTDBL} and
\ac{CQHDBL} updates to the NVMe registers.

\begin{figure}[!ht]
    \centering
    \includegraphics[scale=1.3]{pict/ziti_easy_arch-rq_hdr}
    \caption{Format of the header used on the \acl{RQ} interface}
    \label{fig:pcie_tlp_rq_hdr}
\end{figure}

\footnotetext{The x86\_64 architecture addresses its physical memory space with resolution to bytes.}

The \ac{CQ} header (\Cref{fig:pcie_tlp_cq_hdr}) follows a format similar to the \ac{RQ} header, where
the user logic acts as a PCIe Completer. The \emph{Target function} field indicates to which
Function a request is targeted. When multiple \acp{BAR} are used, the core puts the index of the
specific one in the \emph{BAR ID} field. The size of a targeted BAR is determined by the \emph{BAR
Aperture} which serves as a mask for the address (for example, the value of 12 signifies that the
BAR has a size of 4~KiB and, therefore, the address bits [63:12] can be ignored)\cite{pcie_amd_ip}. Other fields
of this header are either reserved or used to create a response for \ac{MRd}.

\begin{figure}[!ht]
    \centering
    \includegraphics[scale=1.3]{pict/ziti_easy_arch-cq_hdr}
    \caption[Format of the header used on the \acl{CQ} interface]{Format of the header used on the
        \acl{CQ} interface. The yellow fields are used by the DMA Iuventus core itself whereas the
        green ones are only used to create the CC header of the response.}
    \label{fig:pcie_tlp_cq_hdr}
\end{figure}

For posted requests, such as \ac{MRd}, the response is dispatched on the \ac{CC} interface by using
the 3~DW TLP header in~\cref{fig:pcie_tlp_cc_hdr}. The header may be followed by a payload. The
\emph{Address} is a byte-level address of the first byte of the memory block being transferred. The
size of the payload is specified in the \emph{Dword count} field. Because the CQ header contains
only DWord-aligned address, the DMA logic needs to calculate the byte-level offset by reading the
\emph{TUSER}\cite{axis_spec} signal of the AXI-Stream bus on the CQ interface which contains the
\ac{FBE} as a mask of valid bytes in the first DWord of the payload. The byte offset is then
adjusted by factoring initial valid/invalid bytes in the calculation. There is another field used,
namely the \ac{LBE} which is located in the same AXI-Stream signal. This mask serves for the last
DWord of the payload to distinguish valid bytes. An exact byte length of the payload can be
calculated by using the Dword count, \ac{FBE} and \ac{LBE} fields\cite{pcie_amd_ip}.

Each completion transfers a status of either \emph{Successful Completion}, \emph{Unsupported
Request} or \emph{Completer Abort} in the \emph{Status} field. Similar to the Target Function field
in the RQ header, there is an index of a completing Function. Other than that, there are multiple
fields that need to be copied from the Requester TLP header, such as \ac{AT}, \emph{Requester ID},
\emph{Tag}, \ac{TC} and \emph{Attributes} so that the Completion is correctly routed to the
Requester and then correctly classified within its internal system. This applies especially for the
Tag field which marks all Completions for a specific Request. A Completion can be dispatched in
multiple TLPs (especially if the MRRS of the Completer is bigger than the MPS value), which is
controlled using the \emph{Byte count} field. The value indicates the remaining amount of bytes that
are needed to complete a request including the length of the current payload. The last Completion
TLP has the value of \emph{Byte count} equal to the byte-length of its payload\cite{pcie_amd_ip}.

\section{Software configuration}

This thesis uses the \emph{NDK-FPGA}\cite{ndk_fpga_github} framework that serves as a development
platform for datacenter-grade FPGA acceleration cards providing stable communication infrastructure
and allowing to instantiate custom user logic. In order to provide the software control, the
\emph{NDK-SW} package is delivered along with the previous framework\cite{ndk_sw_github}. This
package allows access to internal \ac{C/S} registers of various components within the FPGA logic
from userspace. Additionally, this thesis utilizes the \ac{SPDK} framework in order to correctly
configure an NVMe device and establish a \ac{P2P} communication chain between an accelerator card
and the storage drive. SPDK is used to create high-performance applications where drivers are
running in the userspace\cite{spdk_paper}. This enables fast prototyping without the need to operate
with kernel modules which can cause fatal system failures if not done carefully\cite{kernel_usr_io}.

A device that is normally maintained by a kernel driver can only be accessed by a defined API that
usually utilizes the \emph{sysfs} filesystem or, when used in custom applications, the \emph{ioctl}
calls with defined flags\cite{linux_drivers}. Upon every modification of such driver, it needs to be
compiled and inserted into the kernel, not to mention the limited debugging options. This
mechanism, however, prevents malicious applications from accessing the MMIO resources directly. In
order to move all of the control to the userspace, a device needs to be unbound from a particular
driver and bound to a new driver\footnotemark. The Linux kernel stack provides two "dummy" drivers
that serve the purpose of this thesis. If \ac{IOMMU} is disabled, the system operates only with
physical addresses, the device needs to be bound to the \verb|uio_pci_generic| driver, whereas, if
enabled, the system operates with \acp{IOVA} and the \verb|vfio-pci| driver needs to be used. Both
of these drivers expose devices' BARs to userspace where they can be managed using \ac{MMIO}.
Since the first approach uses disabled IOMMU, the application needs to be run as the
superuser\cite{kernel_usr_io,dpdk_usr_drivers,spdk_usp_drivers, spdk_dma}.

\footnotetext{For concrete steps of how to unbind a device, check the~\cref{chap:spdk_conf}}

\subsection{Memory partitioning}

The~\cref{fig:nvme_init_system} presents a standard customer system with a main NVMe drive and an
instantiated set of queues for every core on the processor. The queues that are created for this
purpose are normally stored in the host memory where 2 queue buffers get allocated for every queue
pair. The data buffers are dynamically generated since values of \ac{PRP1} and \ac{PRP2} can vary
with every dispatched command, therefore conforming to the very principle of the \ac{DMA} where data
do not have to be copied to special intermediate buffers but the pointers can directly reference
user data in the host memory.

Since this work wants to establish the \ac{P2P} communication chain between an FPGA accelerator and
an NVMe drive which bypasses the host system, the mapping of queues as well as copy buffers has been
moved entirely to the PCIe BARs of the accelerator. Such system is depicted
in~\cref{fig:iuv_mem_resources}. Because the mapping of PCIe BARs of the accelerator is constant,
the \ac{PRP1} and \ac{PRP2} fields are always pointing either to the \emph{Read Buffer} (which
contains data that the NVMe needs to fetch as a part of the write command) or to the \emph{Write
Buffer} (which is a reserved space where the NVMe uploads its data as a part of the read command).
These buffers are allocated in BARs~3 and 2 respectively. The queue pair is located in the first two
\acp{BAR} where Submission Queue is located on BAR~0 and Completion Queue on BAR~1. By mapping all
of these data structures to the PCIe device, the copying of user data through host memory is
entirely removed.

\begin{figure}[!ht]
    \centering
    \includegraphics[scale=1.3]{pict/ziti_easy_arch-iuventus_memory_scheme.pdf}
    \caption[Allocated system memory resources]{Overview of allocated memory resources as
    initialized by the SPDK application.}
    \label{fig:iuv_mem_resources}
\end{figure}

To configure this P2P chain through SPDK, both the NVMe drive and the accelerator must be unbound
from their default kernel drivers.

The \emph{NDK-SW} framework provides its own kernel driver in order to operate on the PCIe device
and to access its \ac{C/S} registers.
% The
% unbinding simplifies the configuration of each device which directly exposes its BARs' physical
% addresses and its PCIe Configuration Space to the userspace.
The NVMe device can be entirely unbound since no original kernel driver is used and everything
considering a device initialization is handled by the SPDK framework. On the other hand, a different
approach had to be chosen for the accelerator, namely to create a second PCIe Function on the
device. This allows two drivers in the host system to operate on one device while keeping separate
Configuration Space for each one of them. Having this, the access to the configuration registers of
the FPGA's user logic can be preserved while a custom driver for the P2P communication chain can be
used besides that. Since there is only one PCIe IP per one physical connector on the FPGA chip, all
of the requests pass through the same interfaces to the user logic and have to be split according to the
function index and the BAR ID\cite{pcie_amd_ip}.

\subsection{Queue initialization}

This step relies on the initial configuration of the controller done by the SPDK framework.
Especially, this involves the allocation and assignment of Admin queue-pair whose addresses are
stored in the controller's registers (remember that these registers reside in the PCIe BAR of the
NVMe drive). The NVMe controller has two ways of configuration, either through its registers (works
every time) or through its Admin queue pair (after it gets allocated) using the \emph{Set Features}
command.
The following list describes the specific case for the solution presented in this thesis
and not the general one presented in the specification.

\begin{enumerate}
    \item The \ac{CAP} register indicates the maximum supported size of its queues and the
    requirement if queues need to be physically contiguous.
    \item The host allocates space for the Admin Completion and Submission queues with its preferred
    size and writes these to the \ac{AQA} register. Their base addresses are then written to the
    \ac{ACQ} and \ac{ASQ} registers. The Admin queue pair has an implicitly reserved Queue ID 0.
    \item The host submits a \emph{Set Features} command with the \emph{Number of Queues} attribute
    set to the desired amount of I/O Submission and I/O Completion queues. The controller responds
    with the amount of queues that it supports which can be equal or lower than the number of the
    queues requested.
    \item The Queue ID 1 is reserved for the queue pair residing in the FPGA accelerator. This
    reservation is not only internal to the SPDK framework but also necessary for correct
    registration of queues on the NVMe controller to keep consistent state in the whole stack
    when more queues need to be added\footnotemark.
    \item The host allocates I/O Completion queue in the BAR~1 of the FPGA accelerator and clears
    its content. The clearing is necessary since the DMA Iuventus logic could otherwise process past
    entries that have a \emph{Phase Tag} field set to 1 (which is the initial value that is
    considered valid after the queue initialization).
    \item The \emph{Create I/O Completion Queue} command is submitted in order to register the
    queue on the controller.
    \item The host allocates an I/O Submission queue in the BAR~0 of the FPGA accelerator and
    submits the \emph{Create I/O Submission Queue} command to register this queue on the controller.
    \item The Queue ID 2 is reserved for the queue pair for the host to create and maintain a
    filesystem on the NVMe storage.
    \item The host allocates regions in the host memory that are going to hold its I/O queue pair.
    The Completion queue is cleared in the same way as in the case of the FPGA accelerator.
    \item The queues are registered in the same order as previously, first the Completion queue,
    then the Submission queue.
    \item The system is now ready to operate and write/read raw data to/from the non-volatile
    storage.
\end{enumerate}

The current configuration can be read by three means similar to the manners in which a startup
attributes are written. The BAR~0 of the NVMe controller can be consulted or the \emph{Get Features}
command can be submitted to the admin queue-pair. Moreover, the NVMe controller can provide more
detailed information by submitting the \emph{Identify} command. This command provides a substantial
amount of information with regard to the capabilities of either the controller or a namespace
attached to it. This information is then published int the \emph{Identify Controller} and
\emph{Identify Namespace} data structures in the host memory to be reviewed by the
user\cite{nvme_spec}.

\footnotetext{Remember that the Queue ID is used in the \ac{CQE} as a part of a command's unique
    identifier since multiple I/O Submission queues can share one I/O Completion queue}

\section{Command submission}

The P2P chain between the FPGA accelerator and the NVMe drive has been partially established by
creating the accelerator-owned queue pair. In order to perform the full P2P communication, the host
allocates the BAR~2 and BAR~3 of the accelerator and publishes its physical addresses to the
DMA Iuventus module so the submitted commands to the NVMe target the internal buffers of the FPGA.

The \emph{PRP List} is created in the host memory as part of FPGA device initialization as well.
This list resides in the host memory for every requester regardless if the accelerator or the host
engages in the communication. The values of its elements for the FPGA accelerator are constant (that
is because the mapping of BARs does not change) whereas, for the host side, it is created on the
go\footnotemark. Although the PRP List can be arbitrarily long and thus support arbitrarily long transfers,
the NVMe controller limits this. This limit is
published in the \emph{Identify Controller} data structure in the \ac{MDTS} field. This applies to
the write command as well.

\footnotetext{The API of the SPDK framework already contains built-in functions to dispatch
read/write commands to the NVMe device and the maintenance of the PRP List is, therefore, left to
its own devices.}

\subsection{Read command}

The read command to the NVMe consists of several steps as seen in~\cref{fig:nvme_rd_cmd}. The
command is submitted by writing one \ac{SQE} data structure to the Submission queue and updating the
corresponding \ac{SQTDBL} (in case of the accelerator queue pair, this would be SQ1TDBL, whereas for
the host-side, the SQ2TDBL). Afterwards, the controller sends a \ac{MRd} to the Submission queue to
fetch the command. The requester (either the DMA Iuventus module or the host software) responds with
one or more \ac{Cpl} \acp{TLP}. The controller then proceeds to execute the command. The read command
targets either the accelerator's or the host's \emph{Write buffer}, which is specified in the command's
\ac{PRP1} and \ac{PRP2} fields. The NVMe controller can send the data in one or multiple \acp{MWr}.
Finally, to signal the end of command execution, a \ac{CQE} is written to the Completion queue.
As seen in the same figure, the requester can submit multiple commands which are then processed on
the controller. To inform the controller that all completions have been read from the Completion
queue, the requester updates the value of the corresponding CQHDBL in controller's registers
(similar to the SQTDBL, the accelerator updates the CQ1HDBL value whereas the host the CQ2HDBL
value).

    \centering
    \includegraphics[scale=1.3]{pict/nvme_prot-read.pdf}
    \caption[Read command submission]{Read command submission. The indexing of \acp{SQE} is
    completely arbitrary since there is no ordering of commands on the NVMe controller's side if
    multiple commands are fetched.}
    \label{fig:nvme_rd_cmd}
\end{figure}

\subsection{Write command}

The write command contains additional steps compared to the read command as seen
in~\cref{fig:nvme_wr_cmd}. The NVMe controller fetches the command(s) in a similar manner, by
receiving the \ac{SQTDBL} update, dispatching the \ac{MRd} to the requester and receiving
completions with \ac{SQE} data. Afterwards, as a part of command execution, the controller
dispatches one or multiple \acp{MRd} to the requester's \emph{Read buffer} to fetch the data that
are received consequently in one or multiple \acp{Cpl}. The end of command execution is marked by
the write of a \ac{CQE} with the status of the execution. Similarly, the controller can fetch
multiple commands and process them as a batch, dispatching the completion every time. The requester
then, after processing the \ac{CQE}, updates the corresponding value of \ac{CQHDBL}. Optionally, the
requester can issue a \emph{Flush} command to permanently store the data on the \ac{NVM} which can
be seen in the same figure as well. This command is suitable on NVMe devices that contain a
dedicated write cache. It is usually not necessary to explicitly use this command unless
synchronization between two requesters is required. The flushing can be left on the automatic
mechanism of the controller which also flushes the whole content of a cache when it shuts down.

\section{Hardware implementation}

The created hardware design enables full P2P communication between an FPGA accelerator and an NVMe
device. Its block scheme is depicted in~\cref{fig:iuventus_block}. The architecture transports
data on streaming buses rather than memory-mapped ones. The only exception is the \ac{MI}
interface\cite{ndk_mi_spec} connected to the \verb|Software manager| that contains the \ac{C/S}
registers to configure the DMA module. This operates on memory-mapped chunks of 32~bits. The
streaming interfaces, namely \verb|CQ|, \verb|CC|, \verb|RQ|, \verb|Read Data| and
\verb|Write Request + Data| follow the \ac{MFB} specification\cite{ndk_mfb_spec}.

\begin{figure}[!ht]
    \centering
    \includegraphics[scale=1.0]{pict/dma_iuventus_scheme.pdf}
    \caption[DMA Iuventus block scheme]{DMA Iuventus block scheme}
    \label{fig:iuventus_block}
\end{figure}

Because of the \emph{Straddling} feature on the PCIe IP's interfaces conforming to the AXI-Stream
standard, this protocol had to be arbitrarily extended. This mode splits a transfer\footnotemark\@
into two logical regions where each region can contain a start of a packet. If the first packet ends
in the first region, the second packet can begin in the second region, thus effectively increasing
throughput for packets that align in such a way and, especially, for small packets that fit in one
region. This extension includes adding more subsignals to the \verb|TUSER| while ignoring the
\verb|TKEEP| signal since this only works when only one region is utilized.

\footnotetext{One transfer corresponds to one bus beat, meaning a clock period when both TVALID and
    TREADY are high, i.e. both the Transmitter and the Receiver are ready to transfer
    data\cite{axis_spec}.}

\ac{MFB} has been created to exactly address this issue of dispatching multiple frames (or packets)
in one clock cycle and does not need additional signals to facilitate that. A transfer consists of
multiple \emph{regions}, where each is composed of 1 or more \emph{blocks}. Each block contains then
1 or more \emph{items} and each of the items is n bits long. All of the used interfaces of the PCIe
IP have the same configuration, that is 2 regions, 1 block, 8 items and 32 bits\footnotemark\@ per
item (referred to, in a short form, as the \emph{2,1,8,32} configuration) resulting in 512~bits of
width. MFB consists of following signals:

\footnotetext{Setting the item size to 32 bits is in accord with the PCIe interface operating with
    DWords as the smallest atomic unit.}

\begin{description}
    \item[DATA] containing frame payload.
    \item[META] containing optional metadata to the frame. The metadata can be aligned either with
    a SOF of a frame (default) or with an EOF.
    \item[SOF] indicating which regions of a transfer contain a start of a frame.
    \item[EOF] indicating which regions contain an end of a frame.
    \item[SOF\_POS] Pointer for every region where the start of a frame is located. A frame can be
    started with resolution to \emph{blocks}.
    \item[EOF\_POS] Pointer for every region where the end of a frame is located. A frame can be
    ended with resolution to \emph{items}.
    \item[SRC\_RDY] When high, indicates that a transmitter wants to send data.
    \item[DST\_RDY] When high, indicates the readiness of a receiver to accept data.
\end{description}

The interfaces to the user logic, namely \verb|Read Data| and \verb|Write Request + Data|, follow
the \ac{MFB} specification as well but with a different configuration. Since the highest throughput is
no longer required and the smallest size of a memory block that can be requested is 512~B (i.e. one
\ac{LBA}), the configuration has been set to \emph{1,1,64,8} which can be rewired to the standard
AXI-Stream bus. All of the streaming buses run on 250~MHz clock provided by the PCIe IP and the same
clock is used to drive all other systems of the DMA Iuventus module. The \ac{MI} interface runs on a
slower, 100~MHz clock to relax timing constraints.

\subsection{Software initialization}

The register map for the DMA Iuventus module is listed in~\cref{chap:iuventus_reg_map}. The
component gets configured by the host software by setting the following registers in the
\verb|Software manager|:

\begin{itemize}
    \item \verb|DBL Mask| that serves as a mask of all Doorbell pointers. This mask is determined by
    the size of the queues.
    \item \verb|SQTDBL Base Addr H| and \verb|SQTDBL Base Addr L| that contain the address of
    \verb|SQ1TDBL| in the NVMe BAR.
    \item \verb|CQHDBL Base Addr H| and \verb|CQHDBL Base Addr L| that contain the address of
    \verb|CQ1HDBL| in the NVMe BAR.
    \item \verb|RD Buff Base Addr H| and \verb|RD Buff Base Addr L| that contain the address of the
    read buffer allocated in the accelerator's BAR~3. This address needs to be supplied by the
    configuration software since the PCIe IP does not provide an interface to read values in its own
    BARs of the PCIe Configuration Header.
    \item \verb|RD Buff PRP List Base H| and \verb|RD Buff PRP List Base L| that contain the base
    address of the read buffer's PRP List (this list resides in the host memory).
    \item \verb|WR Buff Base Addr H| and \verb|WR Buff Base Addr L| that contain the address of the
    write buffer allocated in the accelerator's BAR~2. This address needs to be supplied by the
    configuration software for the same reason as the address of the read buffer.
    \item \verb|WR Buff PRP List Base H| and \verb|WR Buff PRP List Base L| that contain the base
    address of the write buffer's PRP List (this list resides in the host memory).
    \item \verb|LBA Num Mask| that serves as a limit of the amount of LBAs that can be transferred
    within a single NVMe command (i.e. the maximum amount that can be assigned to the \ac{NLB}
    field in a \ac{SQE} as specified in the \ac{MDTS} field in the \emph{Identify Controller} data
    structure).
    \item \verb|LBA Space Size H| and \verb|LBA Space Size L| that contain the total amount of
    usable LBAs on the provided Namespace regardless of if some \acp{LBA} have already been written
    in the past.
\end{itemize}

Finally, the Iuventus module is activated by assigning bit~0 in the \verb|Control| register to 1
which is acknowledged by the module setting bit~0 in the \verb|Status| register to 1. The
acknowledgement happens after all internal components are initialized.

\subsection{Read command submission}

The user logic requests a read from the NVMe device on the \verb|Read Request| interface
(\Cref{fig:iuventus_block}). This is achieved by setting the amount of LBAs to read on the
\verb|NVME_RD_REQ_LBA_NUM| input and the starting address of the first LBA on
\verb|NVME_RD_REQ_LBA_PTR|. The valid request is confirmed by setting \verb|NVME_RD_REQ_VLD| to high
which is acknowledged by the \verb|Operation control| component by asserting \verb|NVME_RD_REQ_RDY|.
Because of the hardware constraints explained further, the amount of LBAs that can be requested by
the user logic on the \verb|NVME_RD_REQ_LBA_NUM| is capped to 256 (i.e. this signal has a width of 8
bits).

After \verb|Operation control| acknowledges the request, it checks that the request size with its
starting pointer does not exceed the Namespace size (thus writing beyond the namespace boundary). The
component decides the values of \ac{PRP1} and \ac{PRP2} fields for the command based on the amount
of memory pages that will be transferred. After that, the NVMe Read command is created in the
\verb|SQ| \verb|Cmd| \verb|Dispatcher| from the provided parameters that were previously assigned to the \ac{C/S}
registers. SQ Cmd Dispatcher maintains an internal FIFO of available \acp{CID} that are assigned to
every dispatched command. Reading the status of this FIFO helps to assert the amount of flying
commands that are currently executed/waiting on the NVMe controller. This status is available in the
\ac{C/S} registers as the \verb|TAG FIFO Status| field.

The command is then stored in the \verb|Sub Queue| located in the \verb|PCIe| \verb|Transaction|
\verb|Buffer|. Originally, this buffer has been developed as a part of the DMA Calypte
project\cite{calypte_github} to store incoming \acp{TLP} on the \ac{CQ} interface. The advantage of
this buffer is the ability to store on a link data rate without blocking with address resolution to
bytes while also able to store two individual packets (this stems from the two regions the CQ bus is
divided into where each packet can address a different location). Because the reading side of the
\verb|PCIe Transaction Buffer| has been designed to put data on a streaming bus, this component is
used twice in the Iuventus module even when it is underutilized for the Submission queue/Read buffer
side. The constraint of capping the amount of \acp{LBA} being requested is caused by the maximum
size of the memory array in this component which consists of 64~BRAM blocks (since the bus is
64~bytes wide and byte-level addressability is needed). This array has an overall capacity of
256~KiB and it is split into two sectors where one is occupied by the Submission/Completion queue,
while the other with the Read/Write buffer. The size of each sector corresponds to exactly 256~LBAs;
hence, the maximum read/write requests size is capped at this value.

As soon as the \verb|SQ Cmd Dispatcher| creates the command and stores it in the Submission queue it
sends an update of the \ac{SQTDBL} to the \verb|Doorbell updater|. This queues this update and then
dispatches it whenever the \ac{RQ} interface is ready. After restart or a long period where the
pointer has not been updated, the update is dispatched immediately. After dispatching one update,
the delay counter is activated. The pointer can be further updated during this delay but the update
is dispatched first when the delay expires. This mechanism has been added to enable submitting
multiple commands and then fetching them in batch. This reduces the pressure on the PCIe interface
by lowering the amount of \acp{TLP} containing only synchronization data.

The \acl{MRd} from the NVMe controller arrives on the CQ interface and is routed from the
\verb|CQ Metadata Extractor| to the \verb|PCIe Header FIFO|. The Metadata extractor calculates the
exact byte address and splits TLPs based on the BAR and the request type. The header FIFO receives
all of the read requests that can target either the Read Buffer or the Submission Queue. The read
requests are processed by the \verb|PCIe Read Responder|, which is able to read data of arbitrary
length from the transaction buffer and put them on the \ac{CC} interface. If the size of data
exceeds 128~B, the payload gets segmented into 128-byte chunks\footnotemark\@ which are dispatched
as separate \ac{Cpl} \acp{TLP}, effectively utilizing the \emph{Dword count} and \emph{Byte count}
fields in the \ac{CC} \ac{TLP} header. A single command in a queue is usually read by just one
\ac{MRd} from the NVMe controller.

\footnotetext{Segmenting data into chunks of this size simplifies the design. Otherwise, the
\ac{MPS} value would have to be consulted with the PCIe IP and the size of the segmented blocks
would have to be dynamically configurable since the value of MPS can range from 128~B to 4096~B.
Setting the size to the smallest possible value enables the use of the static setting for
segmentation.}

After some delay, the controller sends data from the \ac{NVM} that had been requested in the
processed command. These data are routed towards and written to the \verb|Write Buffer|. Finally,
the controller sends a \ac{CQE} with the status of the operation. The \verb|CQE Processor| component
detects the received \ac{CQE} based on the current value of the \ac{CQHDBL} and the \emph{Phase
tag}. The component then returns the status of the command to the \verb|Operation control| and
updates the value of \ac{CQHDBL}\footnotemark\@ in the \verb|Doorbell updater|. When the returned
status is successful, the \verb|Operation control| then reads the data from the \verb|Write buffer|,
thus returning them to the user logic.

For software-facing reporting, the command status is returned on the
\verb|Operation| \verb|status| interface. When the \verb|OP_STAT_TYPE| signal on this interface is set
high, the status of the last read request is returned, otherwise, when set to low, the status
of the last write request is returned. For the accurate distinction of status, the
\verb|OP_STAT_CODE| signal outputs the status of the operation, which is listed
in~\cref{tab:op_stat_codes}. A valid status is signalized by setting the \verb|OP_STAT_VLD| output
high for a single clock cycle.

\footnotetext{The values of Doorbell pointers can be found in the registers listed in
\cref{chap:iuventus_reg_map}. The value of \ac{CQTDBL} is not provided since it is internal to the
NVMe controller and it is not necessary for the correct processing of \acp{CQE}.}

\subsection{Write command submission}

A write request is initiated by directly writing a packet on the \verb|Write Request+Data| interface
which is a \ac{MFB} bus with the \verb|META| signal containing a pointer of the LBA to which the
packet data should be written. This pointer travels with the start of every packet. The Iuventus
module checks the length of an incoming packet which is provided for the \verb|Operation control|
after the packet ends while the packet data have been written to the \emph{Read buffer}. The packet
should not be longer than 128~KiB in order to fit in the buffer. If a packet is longer than this, it
gets truncated to fit in and the rest of its data is dropped.

After receiving a packet's length, the \verb|Operation control| checks if the amount of data that
are going to be written to the NVMe on the specified \ac{LBA} does not cross the namespace boundary.
When this check passes, the component submits an NVMe Write command in the same way as the previously
described submission of the read command, i.e. by writing the command to the Submission queue and
then updating the \ac{SQTDBL} in the \verb|Doorbell updater|. The NVMe controller reads that command
from the Submission queue whose data are then returned on the \ac{CC} interface. During execution of
this command, the controller sends multiple read requests to the read buffer that are routed from
the CQ interface through the \verb|CQ Metadata Extractor| to the header FIFO. These requests can
address data chunks as large as the current \ac{MRRS} setting (up to 4~KiB) but the
\verb|PCIe Read Responder| dispatches completions in 128~B chunks as previously described.

The completion of a command happens in the same way as in case of the read command where the NVMe
controller writes a \ac{CQE} into the Completion queue which is then detected by the
\verb|CQE Processor| and its status returned to the \verb|Operation control|. The status is also
signalized on the \verb|Operation status| interface with the same status codes as described
in~\cref{tab:op_stat_codes}. The flushing operation is triggered after a delay of approximately 1~s
to ensure some level synchronization when the sectors are going to be accessed by the host system in
the future. The flushing command is dispatched from the \verb|SQ Cmd Dispatcher| as well with all
its fields set to 0 except the index of a namespace.

\chapter{Filesystem integration}
\label{chap:filesystem}

The NVMe userspace driver provided within the SPDK framework\cite{spdk_usp_drivers} with its API
provides all necessary functions for NVMe device configuration and status
management\footnotemark. The API also provides functions, like \verb|spdk_nvme_ns_cmd_read| and
\verb|spdk_nvme_ns_cmd_write| that can submit read/write NVMe commands and operate on the raw data
stored on the \ac{NVM}. However, operation on the raw sectors of \acp{LBA} cannot be easily used in
any OS since it introduces a large amount of device-specific code. This thesis aims to share a
running (but also standardized) filesystem on the NVMe drive between a host and an FPGA accelerator.

\footnotetext{The driver does not use interrupts since these cannot be routed to the userspace and
they introduce a significant overhead due to forced context switches. Therefore, they have been
disabled for the NVMe drive and for the FPGA accelerator as well. The NVMe standard supports wide
range of interrupt types used in the PCIe. These interrupts can be used to notify the requester that
new entries arrived to the completion queue. The SPDK framework incorporates polling to wait for
devices which helps to reduce latency for the cost of putting some load to the CPU running polling
loops\cite{spdk_usp_drivers}.}

Linux-based operating systems distinguish three types of devices: \emph{character (char)}
devices that operate with their data in streams (like a sound card or serial console),
\emph{network} devices that serve as an interface with callbacks returning/putting packet data;
and \emph{block (bdev)} devices that operate on large blocks of data (usually 512~B or more). Both
the char and block devices are available as file entries in the operating system (i.e. device files
in the \verb|/dev| directory), although the block devices are the only ones available to be read
back and forth as a normal file (meaning the users are able to perform seek on them)\footnotemark.
These devices are the only ones that are available to run a file system\cite{linux_drivers}
because they provide an abstraction over multiple different storage hardware by disposing of a
unified set of operations\cite{spdk_paper}. Various Linux filesystem drivers then attach to specific
block devices.

\footnotetext{There can be exceptions to some char devices but they generally prohibit to perform a
seek to their device files.}

An interesting case occurs when a userspace framework is used to operate on a block device such as
the SPDK application operating on an NVMe drive in case of this thesis. A filesystem can be mounted
only if it is visible by the kernel, which is where \ac{ublk} comes into scene. This is a
high-performance driver that allows attaching a userspace application (called \emph{ublk server}) as
a kernel-based bdev. The \ac{ublk} framework profits from \verb|io_uring| as a recent addition to
the Linux kernel that introduces a fast way of userspace-kernel communication which is fully
asynchronous and utilizes a pair of queues similar to the ones specified in NVMe (that is
\emph{Submission queue} and \emph{Completion queue}). This way allows to submit multiple requests to
the kernel or vice versa and then just wait for the completion of such batch (works as a part of the
asynchronous I/O contrary to the traditional way which uses system calls that are blocking and only
one can be active at a time)\cite{io_uring_whitepaper}. The scheme of how this is configured is
depicted in~\cref{fig:spdk_ublk}. The ublk server asks the ublk driver to create one or more ublk devices
(located in \verb|/dev/ublkb*|). One device of that sort is then formatted as a filesystem and mounted.
Every operation on this filesystem (like read, write or flush) is forwarded through the ublk driver
back to the server where it is processed and completed\cite{spdk_ublk,ublk_linux_kern_doc}.

\begin{figure}[!ht]
    \centering
    \includegraphics[scale=1.3]{pict/ublk.pdf}
    \caption[Initialized SPDK ublk device]{The block scheme of the initialized application proposed in
    this thesis running one ublk device.}
    \label{fig:spdk_ublk}
\end{figure}

The \ac{exFAT} filesystem has been chosen for the demonstration because of its ease of
implementation, ability to store very large files, and support for very large storages. This
filesystem is also supported by the Linux kernel, thus by available tools, like \verb|mkfs|,
\verb|dumpfs| or \verb|lsblk|. The portion of a filesystem that can store user data (the s.c.
\emph{Cluster Heap}), is organized as an array of \emph{clusters} where each consists of multiple
\emph{sectors}, each, in case of the NVMe drive, of a size of one \ac{LBA}\cite{exfat_spec}. The
configuration used in this thesis uses the cluster size of 128~KiB which corresponds to 256~LBAs
(this cluster size is automatically set by the \verb|mkfs.exfat| utility\cite{mkfs_manpage}). Theses
are the distinctive features of this filesystem:
\begin{itemize}
    \item The allocation of clusters is not kept inside the nodes of the \ac{FAT} (as it is in the
    case of the FAT32 filesystem) but in the \emph{Allocation Bitmap} located in the \emph{root
    cluster} of Cluster Heap.
    \item The nodes of the FAT are still 32 bit pointers. This explains the increasing size of a
    cluster in order to encompass large storage media.
    \item A file does not have to use the FAT if such file can be stored in a contiguous string of
    clusters. This is marked by an attribute of the file and the amount of utilized clusters is
    still determined by a size of the file.
    \item The specification allows to encompass storage volumes as large as 64~ZB. However, it
    recommends to not use more than $2^{24}-2$ clusters in the cluster heap which results in a
    maximum capacity of 512~TB given the maximum cluster size of 32~MB\cite{exfat_spec}.
\end{itemize}

\section{Latency measurement}

The latency has been measured for the requests originating in the user logic on the DMA Iuventus
interface by a custom component created in the \verb|USER_CORE| entity (files
\verb|user_core_ent.vhd| and \verb|user_core_test_arch.vhd|). The measurement circuit consists
mainly of a data logger and a latency meter. The latency meter starts measuring from the dispatch of
either a read request or a write request (remember that, at every point in time, there can be only
one flying request) and ends the measurement after the operation is completed, i.e. when
\verb|OP_STAT_VLD| pulses high. The width of a timestamp has been set to 28 bits which allows to
capture delays over 1~\si{\second}. The valid latency value is stored by the \verb|DATA_LOGGER|
component that stores maximum, minimum and a histogram of captured values. The data logger stores
only 22 bits of a timestamp value from the latency meter since this should be sufficient according
to~\cite{spdk_paper,nvme_latency}. There is a detection of the timestamp
overflow that would mess with the values already stored in the data logger in which case the whole
measurement has to be repeated. The histogram consists of $2^{15}$~bins that give it a resolution of
512~ns. The measurement parameters were varied based on multiple variables:

\begin{itemize}
    \item Operation type: read or write
    \item Access pattern: sequential or random addressing
    \item Selection of request sizes in LBAs: 1, 8, 64 or 256 being copied
\end{itemize}

Each measurement has been repeated 50000 times. The sequential addressing starts at address number 0
and advances in increments equal to the size of the request. The random addressing is actually a
pseudo-random sequence created by a LFSR. The results are listed in~\cref{tab:latency-results}. The
maximal and minimal values are the only ones taken exactly from the data logger. The quantiles are
taken from linearly interpolated cumulative distribution function derived from the histogram. The
read operation is a more expensive one since the NVMe drive has to fetch the data from the \ac{NVM}.
An obvious fact occurs that the bigger the request's size, the bigger the latency. Surprisingly
however, the latency increases only sublinearly indicating that there is some optimization for
fetching of large data on the NVMe's side. With regard to the read operation, there has not been
observed any substantial difference between access patterns.

\verb|Region N| where the FPGA accelerator contains the previously initialized 4 \acp{BAR} and the

correctly bound to the \verb|uio_pci_generic| driver as present in the \verb|Kernel driver in use:|
field.

The \verb|fpga_zero_copy| puts out valuable information about the configured parameters. It probes
all unbound NVMe drives in the system but attaches only our chosen one. The BAR addresses of the
NVMe as well as of the FPGA accelerator. The values of BARs are assigned to the FPGA context as
mentioned previously in the implementation chapter, i.e. BAR~0 address to the Submission Queue,
BAR~1 address to the Completion Queue, BAR~2 address for the write buffer and BAR~3 address for the
read buffer. The assignment is followed by allocating \ac{QID} of 1 for the queues in the hardware
and attaching these queues to the NVMe controller. The doorbell register offsets needed to be
calculated from the offset in BAR~0 of the NVMe combined with \emph{Doorbell stride}. These offsets
are also written to the FPGA context, meaning into the configuration registers of the DMA Iuventus
module. The PRP Lists are initialized afterwards and their physical addresses are published to the
FPGA context as well. Lastly, the ublk device is created as mentioned in the output and published as
\path{/dev/ublkb0} with examples of how to format and mount it.

The system can now be mounted and operated as a normally attached drive. To make the created files
permanently stored is made by running the \verb|sync| command which not only flushes the cache on
the NVMe drive but also the catche maintained by the filesystem driver in the kernel.

\begin{lstlisting}[caption={Formatting the created ublk device}]
# Mounting with relaxed privileges for the non-superuser (uid and gid numbers can differ)
$ sudo mount -o uid=1000,gid=1000,umask=000 /dev/ublkb0 /mnt/nvmexp/

# Create some experimentary files
$ cd /mnt/nvmexp
$ echo "NCT rules!" > tst1.txt
$ echo "Heidelberg is a nice town." > tst2.txt
$ mkdir my_dir
$ echo "Hidden file in one directory" > my_dir/tst1.txt

# Creating large contiguous files is generally problematic for exFAT but this may work
$ dd if=/dev/zero of=./large_file.txt bs=1M count=1

# Permanently store the written files on the media
$ sync /dev/ublkb0
\end{lstlisting}

Before killing the running application, the block device needs to be unmounted. The
\verb|fpga_zero_copy| application can be killed by sending the \verb|SIGINT| signal to it which is
normally bound to Ctrl-C keybinding. This triggers a shutdown process of all devices in the chain
with correct disabling of the DMA Iuventus module.
